Measured 2026-06-16, Central US, B1 (Always-On), 200 samples/path, warm pools. Single synthetic record by primary key. Server-side = timed in the app (no network). Client-felt = headless Chromium over a LAN to Central US.

architectureserver p50server p95client p50client p95
Direct → Azure SQL1.1 ms8 ms45 ms85 ms
Direct → Cosmos24 ms37 ms67 ms88 ms
API → Azure SQL21 ms50 ms63 ms107 ms
API → Cosmos40 ms70 ms83 ms122 ms
Server-side, the API tier costs ~16–19 ms (1→21 ms on SQL, 24→40 ms on Cosmos) — consistent across backends, so it's a real measurement, not noise. Add a real browser over the network and a flat ~45 ms floor appears (your distance to the datacenter), yet the worst path's client p95 is still ~122 ms. Either lens: a fraction of a second. Nothing approaches "seconds."
Method notes: The datastore engine matters more than the hop — Cosmos point-reads run ~22 ms above SQL. The only time this ever showed hundreds-of-ms-to-seconds was on the Free (F1) tier under sustained load (CPU throttling) — a tier choice, not the API. On B1 that tail disappears.
