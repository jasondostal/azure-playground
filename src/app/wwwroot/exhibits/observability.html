Exhibit #5 — The App Map draws itself

Every other exhibit already makes calls across App Service, the API tier, SQL, Cosmos, Service Bus, Event Grid and Functions. Nobody was watching. Turn on Application Insights — one resource, one connection string, no agent to install on any host — and the whole topology, call rates, latencies and failures appear from the traffic itself.

What this drives

A transaction here fans across the stack. App Insights captures each hop as a dependency and stitches them with W3C trace context, so the map shows the edges and the end-to-end trace — automatically.
web app ──HTTP──▶ API tier ──▶ Azure SQL
   │                    └──▶ Cosmos DB
   ├──direct──▶ Cosmos DB
   └──publish─▶ Service Bus / Event Grid ──▶ Functions ──▶ Cosmos   (chain = full)

Telemetry lands ~30–90s after a run. Then:
Application Map ▸auto-discovered topology Live Metrics ▸1s stream, no sampling Failures ▸the faulted edges Performance ▸dependency percentiles Logs (KQL) ▸query the raw telemetry
  • On the App Map, node size tracks call volume and the ring turns red where the injected faults hit — click a node → Investigate failures drops straight into the failing dependency.
  • Live Metrics shows the run happening in real time (sub-second), unsampled — leave it open and click Run again.
  • The Logs blade is the differentiator: the same data as KQL. The query pack is in docs/observability-kql.md — custom metric pg.txn.ms and event pg.txn carry outcome/fault dimensions the auto-telemetry doesn't.

The honest pitch vs Dynatrace

For an Azure-native appApplication InsightsDynatrace
Distributed tracing + app mapbundled, zero-config, no per-host agent rolloutOneAgent — excellent, but per-host/DPS licensed
Ad-hoc analyticsKQL over all telemetry, joins, custom metricsDQL / canned dashboards
Cost at rest / low traffic$0 idle, pay per GB ingestedhost-hours / consumption units add up fast
Causal root-cause AIbasic smart detectionDavis AI — genuinely strong
Full-stack infra / OS / multicloudAzure Monitor, Azure-centricdeep host + multicloud coverage

Not a clean sweep, and pretending it is would be the fast way to lose the room. For an Azure-native stack the App Map + KQL + cost story is the strong, defensible case: you get most of the observability value already paid for in the platform, with no agents to deploy. Dynatrace earns its price where you need Davis-grade root-cause or full-fleet infrastructure coverage.
