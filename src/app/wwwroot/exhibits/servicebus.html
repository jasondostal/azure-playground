Exhibit #2 — Service Bus for synchronous reads (a cautionary tale)

Exhibit #1 showed an API tier is cheap. This shows the thing that actually earns the "it'll take seconds" reputation: using a message queue for a synchronous read.

A queue is async decoupling infrastructure. To fake a synchronous reveal through it, you do this:
UI → enqueue request → [broker] → worker receives → reads SQL
   → worker enqueues reply → [broker] → UI receives reply
One in-process call becomes ~4–6 broker round-trips. Add a naive client poll loop and it compounds. Watch.

~7× the API tier at best; naive polling crosses one full second (~18×). And request/reply over a queue with no sessions occasionally loses replies entirely — slow and unreliable.
Why it's the wrong tool: queues exist to decouple producers from consumers for work that can happen later — not to answer a user staring at a screen now. Basic tier can't even do sessions (the proper way to correlate a reply to its request), so you hand-roll it. The latency you see isn't Service Bus being slow — it's the wrong shape for the job. ← compare to Exhibit #1
